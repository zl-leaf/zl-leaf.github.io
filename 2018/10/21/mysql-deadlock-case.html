循环update导致的mysql deadlock分析

Posted on 2018-10-21 | Edited on 2018-12-21

最近工作上发生了一次mysql的deadlock事件，发生死锁的是几个批处理的脚本，没有显示执行transaction，而且是一条一条sql执行的。以往介绍死锁的案例都是使用事务，然后事务中执行多条sql的，所以面对此情况瞬间蒙蔽。不过也借此机会整理与mysql锁相关的基础知识。

循环update导致的mysql deadlock分析

本案例描述

背景

有一个表stat_day，表结构如下

工作中需要启动n个脚本对这个表进行操作，具体操作是脚本中会有一个循环，循环执行一条sql，此sql可能是update或者insert，脚本中没有显示地开启事务，而且update的结果集是不会有交集的，结果却发生了mysql的死锁。

死锁信息

以下是使用show engine innodb status输出信息中，截取的关于deadlock的信息

简单描述：

  • TRANSACTION(1)执行的语句UPDATE `stat_day` SET `test`='1' WHERE `package_code` = 'P0003263' AND `gid` = 'PM000399' AND `cid` = 'C0000215' AND `statdate` = '2018-10-19'等待着GEN_CLUST_INDEX上一些记录的x锁
  • TRANSACTION(2)执行的语句UPDATE `stat_day` SET `test`='20' WHERE `package_code` = 'P0002754' AND `gid` = 'PM000362' AND `cid` = 'C0000215' AND `statdate` = '2018-09-20'拥有GEN_CLUST_INDEX上一批记录的x锁，并且等待着GEN_CLUST_INDEX上另一批记录的x锁
  • 刚好，TRANSACTION(1)中等待的锁在TRANSACTION(2)上，而TRANSACTION(2)等待的另外的锁不知道在谁那里，反正就是WAITING FOR

解决方案

alter table stat_day add key new_index(statdate, gid, cid, package_code)

探究

没有start transaction也会发生锁等待？

InnoDB有一个autocommit的设置，可以通过select @@autocommit查看，开启的时候，可以看做一条sql语句就是一个事务。

一条update的sql有什么锁操作

加锁细节流程请详看解决死锁之路 - 常见 SQL 语句的加锁分析 - aneasystone’s blog，这里针对案例做一下简要的描述。

首先说明以下几个前提：
  • 表的主键是最基础的索引，如果没有显示设置的话InnoDB会自动创建一个则增主键GEN_CLUST_INDEX。
  • 锁是加到索引上的，有锁操作的时候主键是跑不掉的。
    • 扩展一下，InnoDB中，一般锁都是B-Tree结构的，主键的叶子节点是记录的数据，而其他类型的索引的叶子节点记录的是主键的位置，所以加锁的时候主键是跑不掉的。（相关可以搜索聚簇索引）
  • where条件中的字段，即使加了索引，具体用不用是由mysql决定的，可以通过explain来查看执行计划，本案例中4个字段只用了两个字段的索引。
    本案例发生死锁的原因
    了解了前提，那原因自然就出来了。stat_day表没有主键，所以mysql自动创建了GEN_CLUST_INDEX，通过explain UPDATE `stat_day` SET `test`='1' WHERE `package_code` = 'P0003263' AND `gid` = 'PM000399' AND `cid` = 'C0000215' AND `statdate` = '2018-10-19'可以看到使用的是package_code和gid这两个索引，即加锁的时候会先通过package_code或者gid之一（假设是package_code），找到主键(GEN_CLUST_INDEX)，将package_code和GEN_CLUST_INDEX对应的记录加锁，然后剩下的gid找到对应的主键记录，在对gid和对应的GEN_CLUST_INDEX，由此导致了本案例的死锁。
    解决方案的可行原因
    知道原因之后，解决方案就清晰了，由于业务的特点，statdate, gid, cid, package_code这四个字段对应的结果集不会有交集，所以创建了一个新的索引alter table stat_day add key new_index(statdate, gid, cid, package_code)（自嘲一下，应该使用唯一索引更加合适），通过explain查看发现，update语句使用的是新加的new_index索引，因此比较了不同update sql在GEN_CLUST_INDEX上相同记录加锁的情况，从而解决了本次死锁。
